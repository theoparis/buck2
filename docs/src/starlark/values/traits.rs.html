18//! The values module define a trait `StarlarkValue` that defines the attribute of
19//! any value in Starlark and a few macro to help implementing this trait.
20//! The `Value` struct defines the actual structure holding a StarlarkValue. It is
21//! mostly used to enable mutable and Rc behavior over a StarlarkValue.
22//! This modules also defines this traits for the basic immutable values: int,
23//! bool and NoneType. Sub-modules implement other common types of all Starlark
24//! dialect.
25//!
26//! __Note__: we use _sequence_, _iterable_ and _indexable_ according to the
27//! definition in the [Starlark specification](
28//! https://github.com/bazelbuild/starlark/blob/master/spec.md#sequence-types).
29//! We also use the term _container_ for denoting any of those type that can
30//! hold several values.

68/// A trait for values which are more complex - because they are either mutable
69/// (e.g. using [`RefCell`](std::cell::RefCell)), or contain references to other values.
70///
71/// For values that contain nested [`Value`] types (mutable or not) there are a bunch of helpers
72/// and macros.
73///
74/// ## Types containing [`Value`]
75///
76/// A Starlark type containing values will need to exist in two states: one containing [`Value`]
77/// and one containing [`FrozenValue`](crate::values::FrozenValue). To deal with that, if we are defining the type
78/// containing a single value, let's call it `One`, we'd define `OneGen`
79/// (for the general version), and then have the
80/// [`starlark_complex_value!`](crate::starlark_complex_value!) macro
81/// generate `One` and `FrozenOne` aliases.
82///
83/// ```
84/// use allocative::Allocative;
85/// use derive_more::Display;
86/// use starlark::starlark_complex_value;
87/// use starlark::values::Coerce;
88/// use starlark::values::ComplexValue;
89/// use starlark::values::Freeze;
90/// use starlark::values::FreezeResult;
91/// use starlark::values::Freezer;
92/// use starlark::values::FrozenValue;
93/// use starlark::values::NoSerialize;
94/// use starlark::values::ProvidesStaticType;
95/// use starlark::values::StarlarkValue;
96/// use starlark::values::Trace;
97/// use starlark::values::Tracer;
98/// use starlark::values::Value;
99/// use starlark::values::ValueLike;
100/// use starlark_derive::starlark_value;
101///
102/// #[derive(
103///     Debug,
104///     Trace,
105///     Coerce,
106///     Display,
107///     ProvidesStaticType,
108///     NoSerialize,
109///     Allocative
110/// )]
111/// #[repr(C)]
112/// struct OneGen<V>(V);
113/// starlark_complex_value!(One);
114///
115/// #[starlark_value(type = "one")]
116/// impl<'v, V: ValueLike<'v>> StarlarkValue<'v> for OneGen<V>
117/// where
118///     Self: ProvidesStaticType<'v>,
119/// {
120///     // To implement methods which work for both `One` and `FrozenOne`,
121///     // use the `ValueLike` trait.
122/// }
123///
124/// impl<'v> Freeze for One<'v> {
125///     type Frozen = FrozenOne;
126///     fn freeze(self, freezer: &Freezer) -> FreezeResult<Self::Frozen> {
127///         Ok(OneGen(self.0.freeze(freezer)?))
128///     }
129/// }
130/// ```
131///
132/// The [`starlark_complex_value!`](crate::starlark_complex_value!) requires that
133/// the type have an instance for `Coerce`, then the macro defines two type aliases.
134/// ```
135/// # use crate::starlark::values::*;
136/// # #[derive(Debug, Trace)]
137/// # struct OneGen<V>(V);
138/// type One<'v> = OneGen<Value<'v>>;
139/// type FrozenOne = OneGen<FrozenValue>;
140/// ```
141///
142/// To make these aliases public (or public to the crate) pass a visibility
143/// to the macro, e.g. `starlark_complex_value!(pub One)`.
144///
145/// The macro also defines instances of [`ProvidesStaticType`] for both,
146/// [`AllocValue`](crate::values::AllocValue) for both,
147/// [`AllocFrozenValue`](crate::values::AllocFrozenValue) for the frozen one, and
148/// [`UnpackValue`](crate::values::UnpackValue) for the non-frozen one.
149/// It also defines the methods:
150/// ```
151/// # use crate::starlark::values::*;
152/// # use std::cell::RefMut;
153/// # struct OneGen<V>(V);
154/// # type One<'v> = OneGen<Value<'v>>;
155/// impl<'v> One<'v> {
156///     // Obtain a reference to `One` from a `Value`, regardless
157///     // of whether the underlying `Value` is a `One` or `FrozenOne`.
158///     pub fn from_value(x: Value<'v>) -> Option<&'v Self> {
159/// # unimplemented!(
160/// # r#"
161///         ...
162/// # "#);
163///     }
164/// }
165/// ```
166///
167/// ## Different types
168///
169/// If the types are different between the frozen and non-frozen values you can define your own
170/// type specialisations as `type One<'v> = OneGen<Value<'v>>` and `type FrozenOne = OneGen<String>`
171/// and use [`starlark_complex_values!`](crate::starlark_complex_values!) which will provide similar facilities to
172/// [`starlark_complex_value!`](crate::starlark_simple_value!).
173///
174/// ## Other types
175///
176/// The macro [`starlark_complex_value!`](crate::starlark_complex_value!) is applicable
177/// when there is a single base type, `FooGen<V>`, with specialisations
178/// `FooGen<Value<'v>>` and `FooGen<FrozenValue>`.
179/// If you have a type where the difference between frozen and non-frozen does not follow this
180/// pattern then you will have to write instances of the traits you need manually.
181/// Examples of cases where the macro doesn't work include:
182///
183/// * If your type doesn't contain any [`Value`] types, but instead implements this trait for mutability.
184/// * If the difference between frozen and non-frozen is more complex, e.g. a [`Cell`](std::cell::Cell)
185///   when non-frozen and a direct value when frozen.
186pub trait ComplexValue<'v>: StarlarkValue<'v> + Trace<'v> + Freeze
187where
188    <Self as Freeze>::Frozen: StarlarkValue<'static>,
189{
190}
191
192impl<'v, V> ComplexValue<'v> for V
193where
194    V: StarlarkValue<'v> + Trace<'v> + Freeze,
195    <V as Freeze>::Frozen: StarlarkValue<'static>,
196{
197}

199/// How to put a Rust values into [`Value`]s.
200///
201/// Every Rust value stored in a [`Value`] must implement this trait.
202/// You _must_ also implement [`ComplexValue`] if:
203///
204/// * A type is not [`Send`] and [`Sync`], typically because it contains
205///   interior mutability such as a [`RefCell`](std::cell::RefCell).
206/// * A type contains nested Starlark [`Value`]s.
207///
208/// There are only two required members of [`StarlarkValue`], namely
209/// [`TYPE`](StarlarkValue::TYPE)
210/// and [`get_type_value_static`](StarlarkValue::get_type_value_static).
211/// Both these should be implemented with the [`starlark_value`](crate::values::starlark_value)
212/// proc macro:
213///
214/// ```
215/// use allocative::Allocative;
216/// # use starlark::starlark_simple_value;
217/// use derive_more::Display;
218/// use starlark::values::NoSerialize;
219/// use starlark::values::ProvidesStaticType;
220/// use starlark::values::StarlarkValue;
221/// use starlark_derive::starlark_value;
222///
223/// #[derive(Debug, Display, ProvidesStaticType, NoSerialize, Allocative)]
224/// #[display("Foo")]
225/// struct Foo;
226/// # starlark_simple_value!(Foo);
227/// #[starlark_value(type = "foo")]
228/// impl<'v> StarlarkValue<'v> for Foo {}
229/// ```
230///
231/// Every additional field enables further features in Starlark. In most cases the default
232/// implementation returns an "unimplemented" [`Err`].
233///
234/// # Note To Implementors
235/// Any additional methods that are added to this trait also need to be added to the
236/// [`StarlarkValue`] implementation in `crate::values::layout::avalue::Wrapper`. Otherwise,
237/// any implementations other than the default implementation will not be run.
238#[starlark_internal_vtable]
239#[allow(non_upper_case_globals, non_snake_case)] // For generated members.

383    /// Return a hash data for self to be used when self is placed as a key in a `Dict`.
384    /// Return an [`Err`] if there is no hash for this value (e.g. list).
385    /// Must be stable between frozen and non-frozen values.
386    fn write_hash(&self, hasher: &mut StarlarkHasher) -> crate::Result<()> {
387        if Self::TYPE == FUNCTION_TYPE {
388            // The Starlark spec says values of type "function" must be hashable.
389            // We could return the address of the function, but that changes
390            // with frozen/non-frozen which breaks freeze for Dict.
391            // We could create an atomic counter and use that, but it takes memory,
392            // effort, complexity etc, and we don't expect many Dict's keyed by
393            // function. Returning 0 as the hash is valid, as Eq will sort it out.
394            let _ = hasher;
395            Ok(())
396        } else {
397            Err(crate::Error::new_other(ControlError::NotHashableValue(
398                Self::TYPE.to_owned(),
399            )))
400        }
401    }

403    /// Get the hash value. Calls [`write_hash`](Self::write_hash) by default.

406        let mut hasher = StarlarkHasher::new();
407        self.write_hash(&mut hasher)?;
408        Ok(hasher.finish_small())
409    }

507    /// Returns an iterator over the value of this container if this value holds
508    /// an iterable container.
509    ///
510    /// **This function is hard to implement correctly.**
511    /// For example, returning a list from this function is memory violation,
512    /// because the list object acting as an iterator is assumed
513    /// to have the iteration lock acquired.
514    ///
515    /// Consider implementing [`iterate_collect`](Self::iterate_collect) instead
516    /// when possible.
517    ///
518    /// This function calls [`iterate_collect`](Self::iterate_collect) by default.
519    ///
520    /// Returned iterator value must implement
521    /// [`iter_next`](Self::iter_next) and [`iter_stop`](Self::iter_stop).
522    /// Default implementations of these functions panic.
523    ///
524    /// Starlark-rust guarantees that
525    /// * `iter_next` and `iter_stop` are only called on the value returned from `iterate`
526    /// * `iter_next` is called only before `iter_stop`
527    /// * `iter_stop` is called exactly once
528    ///
529    /// So implementations of iterators may acquire mutation lock in `iterate`,
530    /// assume that it is held in `iter_next`, and release it in `iter_stop`.
531    /// Obviously, there are no such guarantees if these functions are called directly.
532    unsafe fn iterate(&self, _me: Value<'v>, heap: &'v Heap) -> crate::Result<Value<'v>> {
533        Ok(heap.alloc_tuple(&self.iterate_collect(heap)?))
534    }
